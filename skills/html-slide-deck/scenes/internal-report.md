# Internal Report Content

Use this reference for content organization in internal reports, leadership reviews, executive summaries, project reviews, and management mechanism decks.

## Communication Positioning

- Optimize each page for "what should the leader notice in the first few seconds?"
- Prefer evidence, structure, and conclusion over decorative explanation.
- Keep visible copy aligned with the user's source text. Do not add explanatory slide text unless requested.
- Use short labels to clarify page role: `Key Point`, `Scenario`, `要点`, `示例`, `面向售前`, or a project-appropriate equivalent.
- Do not make every page equally loud. Use section dividers, principle pages, case pages, and summary pages to create rhythm.

## Content Principles

- One page should answer one clear question.
- Put the conclusion before detailed support whenever possible.
- Use screenshots, tables, examples, or specific facts as evidence.
- Keep commentary short. It should interpret evidence, not repeat the title.
- Avoid long background paragraphs, unsupported judgment, and filler copy added only to balance layout.
- Preserve user-confirmed wording, especially titles, labels, and page-specific decisions.

## Page Type Usage

### Cover

Use for report category, report title, and month/date. Avoid subtitles unless the user provides one.

### Opening Point

Use for a quote, background fact, viewpoint setup, or why this report matters. It can use a person/organization placeholder, but the content must be relevant to the report's central question.

### Center Statement

Use for one large keyword, principle, or conclusion. Keep supporting copy to one sentence and avoid turning it into a normal text page.

### Section Divider

Use when the report changes topic or operating object. The section guide sentence should explain what this section will resolve.

### Three Focuses

Use for three equal priorities, management objects, levers, or dimensions. Do not force three items if the content naturally has two or four.

### Three Cards

Use for three actions, options, risks, response paths, or policy choices. The card internals can vary, but the three cards should remain comparable.

### Two Category Cards

Use as a two-category container, not as a fixed content schema.

Suitable category pairs include:

- Current / target.
- Problem / action.
- Risk / opportunity.
- Before / after.
- Internal / external.
- Category A / category B.

Flexible internals:

- One side can hold a single main point and optional visual.
- One side can hold two or three bullet points.
- Either side can hold a screenshot, metric, list, mini-flow, or small card stack.
- Keep the two categories visually balanced even when their internal content differs.

Avoid hard-coding a project-specific structure such as "left has one point plus image, right has two points." Treat the template as a layout pattern and adapt the internals to the user's actual content.

### Screenshot / Case Page

Use for evidence. Pair real screenshots or reconstructed evidence with concise interpretation. Do not turn it into a decorative image page.

### Management Rhythm

Use for time horizons, stages, cadences, or operating layers. Each card should state the cadence/stage, the core action, and the review object.

### Summary

Use for takeaways that can be repeated after the presentation. Avoid introducing new evidence on the summary page.

## Template Evolution

- Treat template pages as layout patterns, not fixed content schemas.
- Keep the high-level page type and visual system, but adapt internal modules to the user's actual content shape.
- If a project creates a strong new page type, first validate it in that project.
- After validation, remove business copy, real screenshots, project names, and customer-specific details.
- Add the abstracted page type back to the template only when it is likely to recur.
