# HTML Slide Deck

这个 skill 用来创建或修改浏览器里的 HTML PPT，默认产物是一个单文件 HTML。这个 HTML 应该能直接播放，也能在浏览器里做简单编辑，必要时再导出 PNG 或 PPTX。

## 核心流程

1. 严格解析用户给的原文。除非用户要求，不主动增加可见文案。
   - 任何幻灯片中都不得出现面向制作者的元说明文字，例如“适用场景”“风格”“模板使用”“按主题自由设计”“未套用固定模板”“交互说明”“快捷键说明”等；这些只能放在交付说明或注释中，不得作为演示现场可见内容。
2. 新建 HTML PPT 时，写 HTML 前必须先问并拿到三类关键信息，除非用户已经在同一条消息中明确给出：
   - 内容材料：主题、大纲、逐页文案或可用于整理大纲的原始材料。
   - 交付目标：场景/受众/页数或大致篇幅，以及是否需要 HTML 之外的 PNG/PPTX 导出。
   - 模板使用：是否使用内置模板；如果使用，指定 `executive-light`、`presales-light`、`presales-dark`，或允许自动匹配/随机；如果不使用，说明按主题自由设计。
3. 如果接手已有项目，先读交接文件，例如 `AGENT.md`、`README.md`、`工作归档*.md`，再检查当前 HTML。
4. 写 HTML 前，先在内部规划页面：页码、页型、主视觉锚点、布局策略。
5. 根据场景选择最接近的模板和组件。需要借鉴历史经验时，只读 `references/common-corpus-best-practices.md`，只复用抽象结构和版式逻辑，不复用私有业务内容。
6. 生成或修改 HTML。
7. 交付前检查布局、交互、编辑模式和导出效果。

只有在三类开工信息已明确，且内容意图、源材料、风格方向都足够判断时，才不再向用户追问。

## 输入判断

开始做 PPT 前，先判断输入是否完整。

优先识别这些信息：

- 场景：内部汇报、培训课件、售前支持、行业会议、客户支持、展厅介绍等。
- 受众：领导、内部团队、客户、学员、公开听众、展厅访客等。
- 目标：进度汇报、辅助决策、机制解释、培训交付、方案说明、客户展示、参观讲解、复盘总结等。
- 材料：大纲、逐页文案、会议纪要、产品资料、截图、旧 PPT、数据表、参考项目。
- 资产：截图、产品图、logo、图表、品牌规范、参考风格。
- 约束：页数、必用词、禁用词、保密限制、导出格式、是否需要浏览器编辑模式。
- 风格偏好：使用指定模板、随机模板、自动匹配、按主题自由设计、避免像上一次、参考某张图或某个项目。
- 模板使用：当用户没有明确模板时，可以询问是否使用 `executive-light`、`presales-light`、`presales-dark` 之一，或允许随机/自动匹配，也可以允许按主题自由设计，不强制套用现有模板。

判断规则：

- 用户给了完整大纲或逐页文案，就进入模板选择和页面规划。
- 用户只给主题时，先做大纲或问三类开工问题，不要直接写 HTML。
- 用户只给主题但已给场景、页数时，优先追问大纲/内容材料和模板使用；模板使用选项为指定 `executive-light`、`presales-light`、`presales-dark`，随机/自动匹配，或按主题自由设计。
- 用户要求参考旧项目时，只复用抽象风格和结构，不复用私有业务文案或私有资产。
- 页面以截图为核心时，读取 `references/common-screenshot-case-pages.md`。
- 页面是“2-3 个观点 + 1 张主图”的图文证据/案例结构时，如果用户没有提供真实截图、产品图或案例图，但页面需要一个强视觉锚点，可以使用 image generation 生成一张与主题匹配的位图视觉，再嵌入 HTML；不要只放“这里插入图片”这类占位文字。
- 生成图只用于概念视觉、场景示意、抽象能力图、医疗科技氛围图或非真实系统界面。需要证明真实系统、真实客户案例、真实数据或真实产品能力时，不得生成伪截图伪证据；必须使用用户提供的真实材料，或改成观点页、流程页、矩阵页。
- 用户没给风格时，根据受众、目标、内容形态选择最接近的场景和模板。
- 用户明确要求自由发挥或按主题定制时，可以跳出现有模板的视觉限制，但必须沿用现有模板的交互底座：`#overview` 幻灯片索引、`#editToolbar`/`#editFloatToolbar` 编辑工具栏、本机保存、另存共享、快捷键和导出清理逻辑。自由发挥只作用于主题视觉、页面编排和内容组件，不重新发明播放/索引/编辑模式。
- 用户说不要像上一次时，保留场景逻辑，但更换视觉风格、页面节奏或页面组合。

## 必备功能

- 默认生成单文件 HTML，除非用户明确要求多文件项目。
- 默认全视口播放：slide 填满 `100vw x 100vh`，浏览器播放和编辑时不出现侧边黑边。
- 布局使用 `vw`、`vh`、`min(vw, vh)`、固定比例模块和多视口检查，不能只在一个屏幕尺寸上碰巧正常。
- 每页必须有内容密度约束：内容区不得触底、不得超出视口；常规内容页底部至少保留 `4vh` 安全空白，封面/尾页按视觉需要保留安全边界；卡片、表格、流程框不能因为 `height:100%` 或过大的 `min-height` 撑出屏幕，也不能出现大面积空卡片或无信息空白。
- 内容密度要双向审查：常规内容页以主内容区约 `85%` 覆盖为目标，可接受范围为 `80%` 到 `88%`。这里的覆盖必须是有效信息和有效视觉结构，不是空白卡片背景、空表格或空容器外框；少于这个范围时，优先改为数据大字、紧凑卡片、图解、时间线、矩阵或补充有效要点；超过这个范围时，优先拆页、删减或改成分组结构。
- 密度规则主要作用于普通内容页。封面、目录、过渡页、章节页、金句页和封底可以保留模板原有留白与视觉节奏，不按普通内容页 `80%` 到 `88%` 覆盖率硬算；但仍必须有明确视觉中心，不能像未完成页面。
- “强留白”只允许作为封面、目录、过渡页、章节页、金句页、数据大字页和封底的设计手段。普通内容页不得用大段无信息空白制造高级感，必须有清晰主视觉锚点和有效信息覆盖。
- 内容不足时不要强行套大卡片、大表格或三列网格；优先换成大字观点、数据大字、引用/金句、左右图文、时间线、流程图、矩阵、紧凑卡片或图解构图。内容过多时优先拆页，不要压缩到不可读。
- 导出型项目要按 `1920 x 1080` 截图检查；PPTX 仍保持 16:9。
- 默认不要用固定 `1920 x 1080` 舞台再整体缩放。只有用户明确要求锁定 16:9 浏览器播放时才这样做。
- 快捷键：
  - `F`：切换全屏。
  - `S`：打开或关闭幻灯片索引。
  - `E`：切换编辑模式。
  - 方向键、空格、PageUp/PageDown：翻页。
  - Home/End：跳到第一页或最后一页。
- 幻灯片索引：
  - 使用全屏浮层。
  - 左侧显示分组标签。
  - 右侧显示 16:9 缩略图。
  - 点击缩略图后跳转并关闭索引。
- 编辑模式：
  - 文本元素可 `contenteditable`。
  - 主要模块和卡片可拖拽。
  - 选中元素要有清晰蓝色描边。
  - 选中元素附近出现浮动小工具栏，支持加粗、字号增减、蓝/黑/灰文字颜色、隐藏或删除。
  - 主编辑工具栏保持简单：`本机保存`、`另存共享`。
  - 撤销/重做默认用快捷键：`Ctrl+Z`、`Ctrl+Y`、`Ctrl+Shift+Z`。
  - 导出的 HTML 不应保留运行时生成的缩略图、导航按钮、选中描边或浮动工具栏。
- 交互底座一致性：
  - 新项目必须从最终选定的 `templates/executive-light/`、`templates/presales-light/` 或 `templates/presales-dark/` 的 `template.html` 复制播放、索引、编辑、保存和导出相关 HTML/CSS/JS。
  - 交互代码以模板现有实现为唯一来源：`#overview`、`#overviewGrid`、`#editToolbar`、`#editFloatToolbar`、`#nav`、快捷键、缩略图生成、localStorage、本机保存、另存共享、导出清理和浮动工具栏定位都不得重写成另一套。
  - 即使按主题自由设计，也只能替换页面视觉、内容 section 和主题变量；不得为 `S` 索引、`E` 编辑、`F` 全屏、本机保存、另存共享、浮动工具栏、缩略图生成、localStorage 和导出清理重新写一套不同外观或不同行为。
  - 允许调整主题色以适配页面视觉，但控件位置、基本样式、交互行为、按钮文案和命名结构应保持模板体系一致；不要把 `本机保存`、`另存共享`、编辑状态或浮动工具栏从模板既定位置搬到另一侧。
  - 鼠标点击空白区域默认进入下一页或下一步 reveal；点击导航、索引、编辑工具栏、可编辑元素、链接、按钮等交互元素时不得误翻页。
  - 如果模板自带 reveal、滚轮锁定、触屏滑动、Escape 优先级、F5 刷新当前页等交互逻辑，应直接继承模板逻辑，不要重写成简化版。

编辑模式细节见 `references/common-editable-html-deck.md`。

## 页面规划

写或改 slide 前，先判断每页属于什么页型。

常见页型：

- 封面：标题、月份/日期或短副标题。
  - 封面允许两种稳定构图：
    - 左文右图/一侧视觉：标题组靠左，但整体应处在垂直中区，不得顶着上沿或底沿；右侧图片/图形承担主视觉锚点。
    - 全居中封面：标题、副标题水平居中，标题组垂直位置应略高于页面中心，通常让标题组中心落在 `38%` 到 `45%` 视口高度之间；主视觉可在标题后方或下方，但不能把标题挤到顶部。
  - 封面文字较少时，不要只做机械水平居中；必须检查标题组、页码、主视觉和底部留白的整体重心是否像一张完整封面。
- 尾页：结束语，通常是 `谢谢`。
- 章节页：短章节标题或过渡语。
- 原则页：一个重要判断或规则，常用大面积强调区。
- 双卡片/双表页：左右对比、两类对象、两条路径。
- 表格页：表格是主视觉锚点。
- 左视觉右表格：左侧图形或截图，右侧结构化表格。
- 截图/案例页：证据图片加简短解释。
- 流程页：横向或纵向步骤。
- 数据大字页：一个或多个大数字。
- 总结页：收束性的关键结论。

页型容量硬约束：

| Page Type | Maximum / Minimum Content |
|---|---|
| 普通内容页 | 1 个标题 + 3-5 个核心信息块；超过 5 个核心块优先拆页 |
| 双栏/双卡页 | 左右两侧都必须有有效内容或视觉锚点；任一侧有效高度低于主体区 `55%` 时改成非对称图文、紧凑卡片或大字观点 |
| 三卡页 | 每卡至少标题 + 2 个有效信息点，或标题 + 1 个明确指标/图形；否则不要使用大卡片 |
| 表格页 | 表格必须是主视觉，占主体区约 `65%` 以上；否则改成矩阵、摘要卡片或大数字 |
| 截图/证据页 | 截图/证据图必须可被看清，通常占主体宽度 `55%` 到 `70%`，或占主体高度 `60%` 以上 |
| 流程/时间线页 | 3-6 个步骤为宜；少于 3 步改成原则页或双栏页，多于 6 步拆页 |
| 数据大字页 | 1-3 个关键数字 + 每个数字一句解释；不要把小数字塞进普通卡片 |

空白失败判定：

- 主体区域下半部分连续空白超过 `28vh`，且没有截图、图形、数据、时间线或主视觉承接。
- 卡片面积很大，但内部只有一行标题、一句话或三分之二以上为空。
- 表格、截图、流程图像小物件一样漂浮在页面中央。
- 左右分栏一侧明显短，另一侧也没有视觉补偿。
- 页面只有标题和少量正文，却使用普通内容页模板；应改成大字观点、金句页、数据页、章节页或图解页。

页面规划示例：

| Page | Type | Layout Strategy |
|---|---|---|
| 1 | Cover | 大标题 + 右侧背景层 |
| 2 | Two-table page | 中心判断 + 左右均衡表格卡片 |
| 3 | Screenshot/case page | 2/3 截图 + 1/3 解读 |
| 4 | Principle + two steps | 上方原则强调区 + 下方双模块 |
| 5 | Back cover | 延续封面风格 + 结束语 |

## 文件定位

用下面的分工判断内容应该放在哪个文件里。不要在多层文件里重复写同一条规则，除非下层文件只做简短引用。

- `SKILL.md`：入口和总契约。说明什么时候用这个 skill、整体生成流程、必备功能、目录分工，以及什么时候读取 references。
- `scenes/*.md`：场景叙事逻辑。放受众、目标、故事顺序、章节结构、常见页面组合、推荐模板和推荐组件。
- `components/*.md`：抽象页面组件。放组件意图、输入槽位、稳定结构、可变范围、变体、适配规则和反模式。
- `templates/<style>/template.html`：某个视觉风格的可运行起始模板。包含占位内容、完整交互功能和代表性页面。
- `templates/<style>/component-gallery.html`：同一视觉风格下的更多组件变体展示。扩展可复用布局时从这里沉淀。
- `templates/<style>/design.md`：某个模板族的视觉系统。放字体、颜色、间距、页面几何、信息密度和视觉约束。
- `templates/<style>/component-map.md`：抽象组件到具体 HTML 的桥。说明组件对应哪些 section、class、variant。
- `references/common-*.md`：跨场景、跨模板的公共技术手册。放编辑模式、导出、截图页、动画、交付检查、历史经验。不要把场景叙事规则或模板专属视觉规则放这里。

## 三层架构

这个库按三层协作：

- `scenes/`：回答“这个场景怎么讲”。
- `components/`：回答“这一页的信息关系怎么表达”。
- `templates/`：回答“这个视觉风格下具体长什么样”。

生成顺序：

1. 识别场景，读取匹配的 `scenes/*.md`。
2. 根据场景推荐或用户偏好选择视觉模板。
3. 根据每页的信息关系选择 `components/*.md`。
4. 读取模板里的 `component-map.md`，把抽象组件映射到具体 HTML section。
5. 根据实际内容调整布局、层级、密度和视觉重心，同时保留组件的核心信息关系。
6. 如果用户选择自由设计，在第 2 步仍要选一个现有模板作为交互底座，只替换视觉系统和页面内容，不重写底座组件。
7. 如果用户明确指定模板，或随机/自动匹配到某个模板：
   - 封面、目录页、章节/过渡页、封底必须严格使用该模板的对应页面结构、logo 露出、底层背景和交互底座，不得重新设计成另一套。
   - “严格使用模板”指直接复制模板 `template.html` 中对应的完整 `section` 结构和相关 class，再只替换项目标题、日期、目录项、章节名、结束语等业务文本；不要重新写封面标题布局、封底布局、目录页布局或过渡页布局。
   - 模板结构优先级高于密度规则：封面、目录、章节/过渡页、封底保留模板既定留白、标题位置、logo、背景和页面节奏，不用普通内容页容量表重排。
   - 普通内容页必须先做“内容关系 -> 模板现有版式”的匹配；符合模板版式时严格沿用该版式的 section/class/层级，只替换文案和必要数据。
   - 如果模板自带普通内容页版式导致内容明显空洞，必须在继承标题区、logo、页码、导航、索引、编辑和导出逻辑的前提下调整主体布局；不要为了“严格套模板”保留空卡片或漂浮小表格。
   - 只有模板中不存在匹配版式，或内容关系确实会被模板版式扭曲时，才允许新增自定义版式；新增版式仍必须继承模板标题区、logo、页码、导航、索引、编辑和导出逻辑。
   - 模板自带 logo 必须保留。封面、目录、过渡页、普通内容页和封底按模板原有 logo 规则展示；除非用户明确要求去除或替换品牌。

组件使用原则：

- 保留组件的核心信息关系。
- 可以调整比例、层级、视觉重量、模块数量和内部结构。
- 不要为了套组件而改变内容含义。
- 优先做有意义的适配，不要机械复制模板页。
- 先判断内容关系，再选择组件：内容少时优先选 `data-hero`、`quote-statement`、`section-divider`、`process-flow` 或紧凑图解；内容对称时才选双卡/三卡；有证据图时优先选 `evidence-case`。
- 使用 `evidence-case` 且没有真实图片素材时，先判断这张图是否可以是“生成型概念视觉”。如果可以，生成一张与页面标题和 2-3 个观点一致的横向位图视觉，作为主视觉嵌入页面；如果不可以，不要伪造证据图，改用其他组件。
- 组件尺寸要有上下限：正文卡片优先使用 `min-height`/`max-height`、`align-content`、`overflow:hidden` 或压缩文案来控制密度；如果内容不足，不要把空卡片强行拉到接近整页高度，也不要让三分之二以上卡片面积为空；大卡片内部应有分层信息、图标/数字、分隔线、标签、指标或 2-3 个要点支撑，否则改成紧凑卡片或图解结构。如果内容过多，拆页或精简，不要缩到不可读。
- 生成后必须做逐页密度审查：浏览器可用时用脚本逐页读取 `.content` 与其内部文本/图形/表格等有效元素的 bounding box，检查有效信息覆盖是否接近 `85%`，并检查底部安全空白；浏览器不可用时至少逐页人工检查 HTML 结构中是否存在超大空卡片、过小内容组或明显溢出风险。
- 如果某种新布局反复出现，先在项目中验证，再沉淀成组件变体。

## 模板体系

模板按场景推荐和视觉风格选择。一个场景可以推荐多个模板，避免每次做出来都长一样。

已支持的场景：

- 内部汇报：领导评审、管理机制、项目复盘、业务更新。
- 培训课件：课程、内部赋能、流程解释、入职培训。
- 售前支持：解决方案、价值说明、对比说明、提案支持。
- 行业会议：公开演讲、主题分享、行业洞察、数据叙事。
- 客户支持：客户说明、成功复盘、服务报告。
- 展厅介绍：展厅循环播放、导览讲解、产品/品牌介绍。

模板选择流程：

1. 识别场景；如果现有场景不匹配，就定义新场景。
2. 识别内容形态：证据/截图、机制/表格密集、战略叙事、数据大字、对比选择或混合型。
3. 选择最接近的视觉模板。售前支持或行业会议类内容，可以先看 `references/common-corpus-best-practices.md` 里的 light-business 与 dark-technology 区分。
4. 如果选择随机/自动匹配，随机或匹配的是模板外观和组件组合；如果选择自由设计，仍要选择一个现有模板作为交互底座。
5. 指定模板时，模板页不是“参考图”，而是工程和版式来源。封面、目录、过渡页、封底必须从模板对应 section 直接复制后替换文字；内容页优先复用模板现有版式。
6. 模板内容页只是业务文案占位，不是固定业务内容。替换文案时可以调整模块内部结构，但不得改变模板版式的核心空间关系、标题区、logo、导航和交互行为。
7. 项目中出现稳定、可复用的新布局时，先验证，再清理成组件变体，加入对应模板的 `component-gallery.html`，并更新 `component-map.md`。

每个正式可复用模板的文件结构按模板类型决定：

- 通用组件型模板，例如 `executive-light/`，至少保留：
  - `template.html`
  - `component-gallery.html`，如果这个视觉风格有多个变体
  - `design.md`
  - `component-map.md`
- 售前成品型模板，例如 `presales-dark/`、`presales-light/`，当前只保留：
  - `template.html`
  - `design.md`
- 售前成品型模板不得再新增外部 `images/` 目录；如需替换图片，应转为 data URI 内嵌进 HTML。

不要把场景叙事规则放进视觉模板，除非它确实只属于这个模板。可复用场景逻辑放 `scenes/`，可复用页面逻辑放 `components/`，视觉实现放 `templates/<style>/`。

## 布局检查

交付前逐页检查并修正明显问题。

最低检查项：

- 页型是否清楚？
- 是否存在面向制作者的元说明文字；如果有，必须移到交付说明或删除。
- 如果使用模板，封面、目录、过渡页、封底是否严格沿用模板，logo 是否保留？
- 内容页是否优先匹配了模板已有版式；若自定义，是否有“模板没有匹配版式”的理由？
- 基础交互是否继承模板：空白点击下一页、键盘翻页、滚轮/触屏、S 索引、E 编辑、F 全屏、Escape 行为是否完整？
- 主视觉锚点是什么？
- 封面属于“左文右图”还是“全居中”；标题组是否在垂直中区或略高于中心，而不是贴近顶部？
- 封面页码、标题、主视觉是否互相避让，整体重心是否均衡？
- 主视觉锚点是否占据足够空间？
- 有效内容密度是否接近 `85%`：有效文字、图形、表格、标签、数据、分隔线等是否共同形成足够覆盖，而不是只让空容器占满页面？
- 页面上下是否平衡？
- 左右模块是否对齐、可比较？
- 表格、卡片、截图是否过短、过高、太靠上或太靠下？
- 内容模块底部是否仍在视口内，并且常规内容页底部是否至少有 `4vh` 安全空白？
- 空卡片是否过大导致页面看起来像未完成；若卡片内部三分之二以上为空，必须补充分层信息或改为更紧凑的卡片、数据大字、流程、时间线、矩阵或图解构图？
- 每个卡片是否至少有标题 + 2 个以上有效信息点，或承担明确视觉功能；否则不要使用大卡片。
- 内容过多时是否拆页或精简，而不是缩小字体、压缩行距或让模块溢出？
- 内容页标题栏是否统一？
- 背景装饰是否在内容后面，而不是抢内容？
- 如果有动画，动画是否帮助阅读顺序？
- 在导出模式和缩略图模式下，页面是否完整可见？

最终交付检查见 `references/common-delivery-checklist.md`。

## 导出流程

用户要求生成图片版 PPTX 时：

1. 用浏览器截图自动化把每页 HTML 截成 `1920 x 1080` PNG。
2. 确认所有 PNG 都是 `1920 x 1080`。
3. 生成 PPTX，每页放一张满版图片。
4. 用户还在确认 HTML 设计时，不要提前生成 PNG/PPTX。

打包或导出前读取 `references/common-delivery-checklist.md`。
