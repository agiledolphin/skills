# Evidence Case

## 用途

用截图、案例、证据图或真实材料支撑一个判断。

## 输入参数

- `section_label`
- `title`
- `evidence_visual`
- `interpretation_points[]`
- `case_title?`
- `takeaway?`

## 稳定核心

- 证据必须是页面主视觉锚点。
- 解读文字用于解释证据，不要重复标题。
- 截图或案例图应足够大，能被观众看清。
- 如果页面结构是“2-3 个观点 + 1 张主图”，且这张图不是证明真实系统/真实客户/真实数据的证据，可以使用 image generation 生成一张横向位图视觉，作为页面主视觉。
- 生成图应服务页面观点：画面要能承接标题和 2-3 个解读点，适合作为售前汇报中的医疗科技、业务场景、能力架构或应用实践视觉锚点。

## 密度规则

- 截图/证据图通常占主体宽度 `55%` 到 `70%`；如果是上下结构，应占主体高度 `60%` 以上。
- 证据图不能像小图片一样漂浮在页面中央。图片不足够清晰时，优先裁切关键区域、放大局部或拆成多页。
- 解读区建议 2-4 个要点；超过 4 个要点优先拆页或改成“证据 + 分步解读”。
- 没有真实证据图时，先判断图片角色：
  - 如果图片用于证明真实系统、真实客户案例、真实数据或真实产品能力，不得伪造；改用原则页、流程页、矩阵页或数据大字页，或向用户索要真实材料。
  - 如果图片只是概念视觉、场景示意、抽象能力图或氛围图，可以生成一张位图视觉，并明确把它当作“视觉示意”使用，不写成真实截图或真实证据。
- 图片标题、注释和 takeaway 要贴近证据图，不能与图片之间留下大段空白。

## 常用变体

- `left-points-right-screenshot`: 左侧 2-3 个解读点，右侧大截图。
- `screenshot-plus-caption`: 大图为主，下方或侧边一条解释。
- `before-after-evidence`: 两张证据图做变化对比。

## 不适合

- 没有真实证据或示例的观点页。
- 需要讲抽象机制但没有图片支撑的页面。
- 截图过小、过糊、无法投影辨认的材料。
